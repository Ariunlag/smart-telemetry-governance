# Repository Instructions

## Scope and truthfulness

This repository is an early foundation, not a production telemetry-governance system. Documentation must distinguish implemented code, placeholders, planned work, and optional research. Do not imply an infrastructure container is application-integrated merely because Compose provisions it.

## Change boundaries

Keep documentation-only work to Markdown, Mermaid, and instruction files unless a task explicitly authorizes runtime changes. Do not alter runtime code, dependency files, Docker/environment configuration, schemas, migrations, tests, or CI for documentation realignment.

## Architecture direction

Prioritize stream discovery, schema/metric/unit governance, confidence-aware classification with abstention, quality, provenance, human review, reproducible evaluation, hardening, and independent pilot validation. Keep duplicate detection, clustering, RAG, agents, causal inference, federation, generic anomaly detection, and knowledge graphs out of the immediate critical path.

## Documentation checks

Before committing, run `git diff --check`, inspect the full diff, verify changed paths, and ensure Mermaid diagrams label current, next, and optional states. Never add private partner information, immigration material, legal strategy, or unproven public-impact claims.
