# Milestones

Each milestone is proposed unless marked complete in [PROGRESS](../PROGRESS.md). Database and API entries describe planned changes, not existing interfaces.

## R0 Repository truth and engineering foundation
**Objective:** establish a truthful baseline and delivery controls. **Scope:** architecture, ownership, decision, evidence, security, and readiness documentation; CI/test conventions. **Non-scope:** runtime features. **Likely files:** `README.md`, `docs/**`, future test/CI configuration. **Database/API impact:** none. **Tests:** documentation link and Mermaid rendering checks. **Edge cases:** stale diagrams and provisional infrastructure mistaken for integration. **Acceptance:** all implemented, planned, and optional components are explicitly separated; zero runtime files changed. **Recovery:** revert documentation commit if facts are found wrong. **Evidence:** traceable inventory and decision record.

## R1 Durable MQTT-to-stream-catalog vertical slice
**Objective:** capture MQTT observations into a durable stream catalog. **Scope:** source registration, authenticated subscription, bounded sampling, stream identity, and catalog query. **Non-scope:** multi-protocol adapters, automated semantics, and long-term raw-data storage. **Likely files:** future MQTT adapter, catalog service/repository, source/stream routes, integration tests. **Database impact:** source, subscription, stream, observation-window, and ingestion-run records. **API impact:** source CRUD, subscription control, stream list/detail. **Tests:** broker-backed integration, idempotency, reconnect, authorization. **Edge cases:** retained messages, wildcard topics, clock skew, malformed payloads, replay. **Acceptance:** a restart-safe catalog records source/topic/schema observations with at-least-once-safe identity and queryable lineage. **Recovery:** disable subscription, retain run evidence, reprocess bounded samples. **Evidence:** discovery coverage and ingestion reliability report.

## R2 Schema, metric, and unit governance
**Objective:** govern observed structure and measurement meaning. **Scope:** schema versions, field mappings, metric vocabulary, unit validation/conversion proposals, and human overrides. **Non-scope:** universal ontology or automatic destructive conversion. **Likely files:** future governance models/services/routes/rules/tests. **Database impact:** immutable schema observations, mappings, metric/unit definitions, versioned decisions. **API impact:** schema/metric/unit review and effective-mapping endpoints. **Tests:** versioning, incompatible changes, unit ambiguity, rollback. **Edge cases:** mixed units, arrays, nulls, locale formats, unknown metrics. **Acceptance:** every approved mapping has versioned evidence; invalid or ambiguous units are flagged or abstained. **Recovery:** supersede mapping; preserve prior decision and recompute derived views. **Evidence:** mapping accuracy and reviewer-agreement dataset.

## R3 Confidence-aware semantic classification
**Objective:** propose telemetry semantics safely. **Scope:** taxonomy, classifier/rules, calibration, abstention, explanation evidence, and review queue. **Non-scope:** forced labels, autonomous operational action, RAG. **Likely files:** future classifier, policy, review APIs, evaluation fixtures. **Database impact:** classification candidates, model/rule versions, confidence, abstention and decision events. **API impact:** classify, queue, approve/reject, audit endpoints. **Tests:** calibration, threshold behavior, deterministic replay, access control. **Edge cases:** unseen labels, conflicting evidence, distribution shift, low coverage. **Acceptance:** held-out selective-risk and calibration targets are met; below-threshold results abstain. **Recovery:** withdraw model version and restore last approved decisions. **Evidence:** Paper 2 benchmark artifact.

## R4 Data-quality and provenance assessment
**Objective:** expose quality risk and lineage. **Scope:** completeness, timeliness, consistency checks; provenance events; evidence links; remediation status. **Non-scope:** generic anomaly detection or causal claims. **Likely files:** future quality rules, provenance store/service, reports, tests. **Database impact:** append-only provenance events, quality assessments, rule versions, remediation records. **API impact:** quality summaries/details and lineage export. **Tests:** late data, missing windows, rule-version replay, tamper detection. **Edge cases:** unavailable clocks, sparse streams, backfill, deleted credentials. **Acceptance:** every assessment links source observations, rule/model version, and timestamp; replay reproduces result. **Recovery:** invalidate derived assessment and rerun from preserved evidence. **Evidence:** issue-detection and evidence-completeness reports.

## R5 Human-review workflow
**Objective:** make governance decisions accountable and usable. **Scope:** role-based queues, assignments, decision rationale, conflict handling, escalation, and audit export. **Non-scope:** autonomous approval or external partner workflow. **Likely files:** future review UI/routes/services/audit tests. **Database impact:** review tasks, assignments, decisions, comments, immutable audit events. **API impact:** queue, claim, decide, reopen, export. **Tests:** authorization, concurrent claim, supersession, audit immutability. **Edge cases:** reviewer absence, conflicting reviews, expired assignments, PII in comments. **Acceptance:** all material changes require attributable decision events and can be reconstructed by audit export. **Recovery:** reopen task and supersede—not overwrite—decision. **Evidence:** reviewer agreement, latency, and correction-rate report.

## R6 Reproducible benchmark and evaluation
**Objective:** make governance claims repeatable. **Scope:** dataset cards, versioned splits, baselines, evaluation harness, result manifests. **Non-scope:** undisclosed proprietary-data claims. **Likely files:** `benchmarks/`, `datasets/`, future evaluation scripts/docs. **Database impact:** optional experiment metadata only. **API impact:** none required. **Tests:** fixed seeds, schema validation, metric regression. **Edge cases:** label leakage, data licensing, class imbalance, unavailable artifacts. **Acceptance:** a clean environment reproduces published metrics within declared tolerance. **Recovery:** retract invalid run and preserve corrected manifest. **Evidence:** Paper 1–3 artifacts.

## R7 Production hardening
**Objective:** prepare the validated core for controlled deployment. **Scope:** authn/authz, secrets, encryption, observability, rate/size limits, backups, disaster recovery, SLOs. **Non-scope:** production-readiness claim before controls and exercises pass. **Likely files:** future runtime/config/deployment changes and operations docs. **Database impact:** backup/restore and retention policies. **API impact:** health/readiness and audit access controls. **Tests:** security, load, restore, failure injection. **Edge cases:** broker outage, credential rotation, partial write, clock failure. **Acceptance:** readiness checklist passes and restore/failover exercises meet stated objectives. **Recovery:** documented rollback, backup restoration, and credential revocation. **Evidence:** controlled hardening and exercise reports.

## R8 Independent pilot validation
**Objective:** test the framework with independent operational users. **Scope:** protocol, data agreement, deployment plan, baseline, success criteria, independent analysis. **Non-scope:** claims of broad national importance or cross-site federation. **Likely files:** pilot protocol, deployment configuration, anonymized aggregate reports. **Database impact:** site-local governed records under retention agreement. **API impact:** scoped tenant/site access only if required. **Tests:** protocol dry run, access review, acceptance tests. **Edge cases:** withdrawal, data-sharing limits, site outage, taxonomy mismatch. **Acceptance:** preregistered outcomes and limitations are independently reported. **Recovery:** revoke access, return/delete data per agreement, archive evidence. **Evidence:** Paper 4 data and report where permitted.

## R9 Optional knowledge-graph projection
**Objective:** evaluate a projection only for validated operational graph queries. **Scope:** query inventory, projection design, consistency checks, cost/benefit evaluation. **Non-scope:** Neo4j or graph infrastructure as a default dependency. **Likely files:** future ADR, projection worker, query tests. **Database impact:** derived, rebuildable projection only. **API impact:** read-only validated graph-query endpoints. **Tests:** projection replay, consistency, query usefulness. **Edge cases:** stale projection, cardinality growth, misleading relationship inference. **Acceptance:** named user queries outperform catalog alternatives on defined measures. **Recovery:** disable projection and rebuild from authoritative records. **Evidence:** query evaluation report.

## R10 Optional research modules
**Objective:** evaluate deferred capabilities only after core/pilot evidence identifies a need. **Scope:** bounded experiments for duplicates, clustering, RAG, agents, causal inference, federation, or anomaly detection. **Non-scope:** inclusion in the production critical path without an approved ADR. **Likely files:** isolated experiment directories and ADRs. **Database impact:** experiment-owned, disposable stores. **API impact:** no production endpoints by default. **Tests:** reproducibility, safety, and comparison to core baseline. **Edge cases:** hallucination, automation overreach, privacy leakage, distribution shift. **Acceptance:** an ADR documents user need, measurable gain, risks, and exit criteria. **Recovery:** retire experiment and delete derived data per policy. **Evidence:** negative or positive experiment report.
